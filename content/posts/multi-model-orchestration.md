## 一个反直觉的结论

最近在开发者社区里反复出现一种叙事：

> 一个**不会自己答题**的小模型，却能在复杂任务上打赢单独上场的 frontier 模型。

它不是更会写代码，而是更会**分工**——读任务、拆步骤、指定「谁来规划、谁来执行、谁来验收」，循环直到检查方签字。

这听起来像管理学鸡汤，但落在 AI agent 上，对应的是非常硬的工程问题：

- 单一模型有**固定盲区**；
- 让模型**自检自己的输出**，往往等于用同一套偏见盖章；
- 「最强」不等于「最适合当前这一步」。

真正值钱的，可能不是再叠一层参数，而是**路由与编排**。

## 为什么「最聪明的人当经理」反而变差

社区讨论里有一个更刺耳的观察：把顶尖大模型塞进「经理」座位，效果有时**不如**一个专司分工的小模型。

这并不神秘：

| 角色 | 需要的能力 | 过度聪明的副作用 |
|------|------------|------------------|
| 规划 | 拆目标、定接口、选工具 | 过度规划、自我加戏 |
| 执行 | 领域强度、工具调用 | 钻牛角尖、忽略约束 |
| 验收 | 找漏洞、拒绝「看起来像对的」 | 和执行共享同一套幻觉模式 |

当「经理」自己也能干活时，它更容易**既当运动员又当裁判**——这正是生产环境里最危险的模式。

## 可落地的四条工作习惯

不必等某篇论文变成标配产品，你现在就可以改流程：

### 1. 默认「模型组合」，而不是「唯一本命」

写代码、查文档、润色文案、做安全审查——拆开分配。  
成本往往更低，盲区更少。

### 2. 强制异模型验收

生成用 A，审查用 B。  
哪怕 B 更弱，只要**错误模式不同**，就能抓到 A 自己看不见的问题。

### 3. 在「交付」前加一个廉价的 Check 步

- diff 是否只改该改的？
- 有没有编造不存在的 API？
- 边界条件有没有测？

这一步消灭的低级事故，通常远超你花在上面的 token。

### 4. 把「会问谁」当成核心技能

未来的杠杆，可能不在「我会写 prompt」，而在：

> 我是否清楚：这一步该用什么模型、什么工具、什么验收标准。

这很像带团队：不会所有乐器的指挥，仍然可以排出好交响。

## 和「个人博客 / 小产品」有什么关系

我们自己做 Web001-Blog 时也踩过类似坑：

- 指望一个端到端方案通吃静态站 + 登录 + 写文件 + SEO；
- 结果在 Vercel 只读文件系统上撞墙，最后用 **GitHub API 写内容 + 构建部署** 才闭环。

这不是「换一个更强的托管」能解决的，而是**换架构角色**：

- 运行时负责读与鉴权；
- Git 负责持久化与审计；
- CI/CD 负责发布。

Agent 编排，不过是同一思想在模型层的重演。

## 结语

模型发布节奏会越来越快，名字会越来越难记。  
更稳的护城河是：

**设计可替换的分工，而不是绑定某一个神话般的大脑。**

下次你想「全部丢给最强模型」时，可以先问一句：

> 这一步，谁来做最合适？谁来验收最不共谋？
