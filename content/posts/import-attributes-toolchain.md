## 一条看起来很小的合并

社区里有一条偏「工程向」的好消息：与 **Import Attributes**（以及围绕 JSON/模块类型断言的能力）相关的能力，正在进一步进入 Node 主干。

单看语法，它并不性感。  
但放在整条工具链上，它属于那种**会改变默认假设**的改动。

## 为什么这比「又一个新语法」重要

现代前端/全栈项目，模块边界往往被五六层工具解释：

```
源码 → TS → Bundler → 运行时 → 测试运行器
```

每一层对「怎么声明这个模块是 JSON / 特殊类型」若不一致，就会出现：

- 本地能跑、CI 挂；  
- Node 能跑、浏览器不能；  
- Bun/Deno 已经支持、Webpack 还要插件。

Import Attributes 的价值，在于推动**声明方式标准化**，减少「每个运行时一套方言」。

## 一张心态表

| 角色 | 短期心态 | 中期动作 |
|------|----------|----------|
| 应用开发者 | 继续用现有打包配置，别急着炫技 | 关注目标 Node LTS 是否覆盖 |
| 库作者 | 双轨导出，别只赌最新语法 | 在 engines / 文档写清最低版本 |
| 基建维护者 | 跟进测试矩阵 | 把 attributes 放进 e2e 样例 |

## 实践建议（务实版）

### 1. 新语法进业务，要比进博客慢半拍

博客可以写「潮流已经来了」；  
生产要以**你服务端实际 Node 版本**和**客户运行时**为准。

### 2. 用「能力探测」而不是「用户代理猜」

能 `try/catch` 或 feature-detect 的，就不要写死「Node 22+ 一定怎样」。

### 3. 文档里写清两条路径

- 推荐路径：现代运行时 + 标准 attributes  
- 兼容路径：loader / 预编译 / 打包内联  

库如果只给一条路径，Issue 会替你写第二篇文档。

## 和 AI 工具链的一点联想

AI coding agent 很爱生成「最新语法」——因为它训练语料里新东西更显眼。  
你若没有 engines 约束和 CI 矩阵，agent 会把仓库推向**演示环境能跑、生产环境尖叫**的状态。

所以：工具链统一，不仅是语言委员会的事，也是**人机协作时的安全带**。

## 结语

Import Attributes 合进 Node，不是终点，而是信号：

> 语言表面的小步，往往在逼整条供应链对齐。

关注它的正确姿势，不是立刻重写所有 import，  
而是检查：你的 **TS / 打包 / 部署镜像 / 文档** 是否还活在五个互相打架的世界里。
